# 采样算子（A3）技术文档

## 1. 在系统中的定位
采样算子是视频增强链路中的基础降维模块。它接收 A1 输出的片段，在不进行复杂内容理解的前提下，以低成本方式从每个片段中选出少量代表帧，作为后续文本抽取、空间聚焦和标注生成的输入。

如果说划分算子解决的是“在哪些时间区间上工作”，那么采样算子解决的是“每个时间区间里到底看哪些帧”。

## 2. 文献依赖等级

- 文献依赖等级：`P2 论文原生`
- 最新支撑强度：`强`

### 2.1 文献锚点
- Video-ChatGPT：多帧视频输入的基础采样范式。
- Video-LLaVA：以固定或均匀采样作为基础数据入口。
- LLaVA-NeXT-Interleave：强调多图/多帧交错输入的稳定性。
- Adaptive Keyframe Sampling：明确把基础 sparse keyframe 作为对照基线。

### 2.2 从文献保留了什么
- 均匀采样依然是不可替代的基线。
- 多帧但低成本的输入方式适合大规模构数。
- 后续所有自适应采样方法都需要一个简单、可控、可复现的起点。

## 3. 子模式定义

### 3.1 CENTER
对每个片段抽取中心帧。

适合：

- 镜头内部高度稳定
- 只需要快速概览
- 后续还会跑 OCR 或 caption

### 3.2 UNIFORM
按均匀时间间隔提取 N 帧。

适合：

- 轻量长视频概览
- 保留粗粒度时间变化
- 作为 adaptive 采样前的基线结果

### 3.3 HEAD_TAIL
提取首帧和尾帧。

适合：

- 关注前后状态差异
- 片段内部变化相对单峰

### 3.4 STATE
在首尾基础上允许轻微容错偏移，尽量采到稳定状态帧而非瞬时模糊帧。

适合：

- 门开关
- 物体状态变化
- 操作前后对比

## 4. 输入规范
采样算子消费 `VideoPacket.segments`，追加 `VideoPacket.samples`。

### 4.1 必需输入
- `video.video_uri`
- `segments[].segment_id`
- `segments[].timestamp_range`
- `segments[].frame_range`
- `config.sampling_mode`

### 4.2 可选输入
- `config.num_frames`
- `config.padding_frames`
- `config.target_fps`
- `config.max_frames_per_segment`

## 5. 输出规范
每个采样结果写入 `samples[]`，至少包含：

- `sample_id`
- `segment_id`
- `timestamp_sec`
- `frame_index`
- `sample_type`
- `resource_refs.frame_uri`
- `confidence`
- `status`

推荐附加字段：

- `relative_position`
- `sampling_mode`
- `is_representative`
- `resolution`

## 6. 关键参数

| 参数 | 作用 |
|---|---|
| `num_frames` | 均匀采样的帧数 |
| `padding_frames` | 首尾采样的容错范围 |
| `max_frames_per_segment` | 控制片段内上限 |
| `target_fps` | 需要按目标帧率抽样时使用 |

## 7. 与上下游的绑定关系

### 7.1 上游
- A1 划分算子提供 `segment_id` 和边界
- A2 片段清洗算子可先过滤不应进入采样的片段

### 7.2 下游
- A5 去重算子可进一步压缩采样结果
- A6 文本抽取算子可直接对样本帧执行 OCR
- A7 空间聚焦算子和 A8 时序定位算子都基于采样结果继续工作

## 8. 典型失败模式

### 8.1 采样过稀
只看到了静态背景，错过真正关键动作。

### 8.2 采样落在模糊帧
尤其在剧烈转场或快速运动中，中心帧不一定最优。

### 8.3 采样数量失控
对长片段仍保留过多帧，后续算子成本过高。

## 9. 推荐使用场景
- Web 短视频基线：`CENTER`
- 长片段粗概览：`UNIFORM`
- 状态变化对比：`HEAD_TAIL` 或 `STATE`
