# 高自由度视频算子系统总览

## 1. 文档定位
本文件是 `算子系统/` 目录下的总设计文档，用于回答四个核心问题：

1. 这套系统为什么不再以“后验 grounding 回看链”为默认路线，而改成“结构骨架优先”的数据构造链。
2. 为什么当前版本固定为 10 个视频算子。
3. 哪些算子对应文献中的强模式，哪些算子是将文献流程系统化落地所必需的中间节点。
4. 这条链路为什么更适合你当前强调的数据构造任务。

## 2. 设计目标

### 2.1 面向数据构造而不是单次推理
当前系统的默认落脚点不是在线问答，而是：

- 自动构造 `Question / Answer / Evidence`
- 保留可追溯的结构骨架
- 让最终样本可以被筛查、回修和复用

因此，系统默认路线不再是“先问题，再回视频找答案”，而是：

`视频 -> 结构骨架 -> Q/A/E 共同合成 -> 回修 -> 质量筛查`

### 2.2 覆盖完整视频数据构造链路
当前落地版的 10 个算子覆盖如下默认主链路：

`A1 划分 -> A2 上下文编排 -> A3 采样 -> A4 粗Caption -> A5 时序结构抽取 -> A6 文本辅证抽取 -> A7 结构骨架构建 -> A8 Q/A/E共同合成 -> A9 一致性回修 -> A10 质量筛查`

这条链路对应的高层流程是：

`segment -> context -> sample -> coarse description -> temporal/text scaffold -> structural scaffold -> joint QAE synthesis -> revision -> screening`

### 2.3 强调共同合成而不是后验补证
当前主流数据构造论文中更常见的做法并不是：

- 先自由生成 QA
- 再回到原视频里做一次严格的 post-hoc grounding

而是：

- 先把视频组织成 narration chunk、时间结构、事件树或图结构
- 再从这份结构骨架中同时导出 Question、Answer 和 Evidence

因此，当前系统把“结构骨架构建”和“Q/A/E 共同合成”提升为默认主链路的核心。

### 2.4 与文献对齐
系统不追求把某篇论文原封不动封装成黑盒，而是把文献里稳定出现的能力拆成可复用的算子。

## 3. 10 个视频算子

| 编号 | 算子名称 | 核心职责 | 文献依赖等级 | 最新支撑强度 |
|---|---|---|---|---|
| A1 | 划分算子 | 把原视频切成时序一致的片段 | P1 论文拆解 | 中 |
| A2 | 上下文编排算子 | 把离散片段整理成可消费的上下文单元 | P1 论文拆解 | 中 |
| A3 | 采样算子 | 统一基础采样与预算感知采样 | P2 论文原生 | 很强 |
| A4 | 粗Caption算子 | 生成片段级或上下文级粗描述，为骨架构建提供语义底座 | P1 论文拆解 | 中 |
| A5 | 时序结构抽取算子 | 抽取事件、时间窗、阶段关系和多跳时序线索 | P2 论文原生 | 很强 |
| A6 | 文本辅证抽取算子 | 汇聚 OCR、字幕、ASR 等文本信号 | P1 论文拆解 | 中 |
| A7 | 结构骨架构建算子 | 把 caption、时序结构和文本辅证组织成 chunk、tree、graph 或其他骨架单元 | P1 论文拆解 | 中强 |
| A8 | Q/A/E共同合成算子 | 在骨架约束下联合生成 Question、Answer 和 Evidence | P2 论文原生 | 很强 |
| A9 | 一致性回修算子 | 对 Q/A/E 三元组做字段对齐、去冲突和局部修订 | P1 论文拆解 | 中强 |
| A10 | 质量筛查算子 | 依据规则、评分和样本价值做最终筛查 | P2 论文原生 | 中强 |

## 4. 为什么是这 10 个

### 4.1 没有再保留独立的空间 grounding 主算子
这是一个刻意取舍。

原因是：

- 当前主流数据构造文献里，最成熟的是 `Q + A + temporal evidence` 的共同合成
- 显式空间框/掩码更多出现在推理或后续增强场景
- 在默认数据构造主链里，空间线索通常被折叠进 caption、事件结构或骨架字段，而不是稳定作为单独主算子

这不意味着空间信息不重要，而是意味着它目前更适合作为骨架字段或后续增强分支，而不是默认主链的独立算子。

### 4.2 为什么要把“结构骨架构建”和“共同合成”拆开
如果把两者混在一起：

- 结构可复用性差
- 不便于后续重采样或重生题
- 也不方便做回修和筛查

因此当前系统把：

- A7：构建 scaffold
- A8：利用 scaffold 合成 Q/A/E

明确拆开。

### 4.3 为什么要单列 A9 和 A10
A8 输出的只是候选 triplets，不应直接出站。

因此需要：

- A9 做一致性回修
- A10 做最终筛查

## 7. 系统设计约束

### 7.1 统一中间态
所有算子都必须通过同一份 `VideoPacket` 交互。

### 7.2 追加式写入
算子只允许追加新对象和新状态，不允许覆写上游原始结果。

### 7.3 共同合成必须依赖结构骨架
A8 不允许绕过 A7 直接对原视频自由生成 Q/A/E。

### 7.4 最终样本必须经过回修与筛查
A8 输出的是候选样本，不是最终数据；必须经过 A9 和 A10 才能出站。

## 8. 当前版本的取舍
这套设计刻意没有把调度器、缓存、分布式执行器纳入 10 个算子，因为这些属于基础设施而不是视频数据构造算子本身。

当前版本也没有把“人工审核”单独拆为第 11 个默认算子，而是把它视为 A10 之后的可选出口。第一版的重点是先把自动化结构骨架和共同合成主链跑通。
