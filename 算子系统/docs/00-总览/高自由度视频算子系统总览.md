# 高自由度视频算子系统总览

## 1. 文档定位
本文件是 `算子系统/` 目录下的总设计文档，用于回答三个核心问题：

1. 这套系统为什么不是简单地堆更多工具，而是围绕一条标准数据流来组织。
2. 为什么当前版本固定为 9 个视频算子，而不是继续无限扩张。
3. 哪些算子是基于文献强驱动的，哪些算子是为了把整条增强链路真正跑通而保留的系统支撑层。


## 2. 设计目标

### 2.1 高自由度
这套系统同时支持两种使用方式：

- 原子化自由拼装：研究者可以按任务需要自由组合算子。
- 预设模板复用：可以快速复现若干论文风格或场景风格的处理链路。

采用此设定的原因，是因为当前 data-centric 文献并不在同一个位置做“视频数据增强”。有些工作把增强前置到 `shot / frame sampling`，目标是提升长视频理解中的关键信息保留；有些工作把增强前置到 `segment split + merge`，目标是先构造语义一致的 event 片段；还有一些工作把增强落在 `time-aware captioning`、`temporal grounding`、`object-centric captioning` 或 `quality selection` 上。换句话说，大家都在“做更好的视频数据”，但切入点并不相同。

截至 `2026-04-01`，能够直接支撑这一判断的代表性工作至少包括：

- 长视频理解与语义采样：
  [Adaptive Keyframe Sampling](https://arxiv.org/abs/2502.21271) / [GitHub](https://github.com/ncTimTang/AKS)，
  [TimeChat](https://arxiv.org/abs/2312.02051) / [GitHub](https://github.com/RenShuhuai-Andy/TimeChat)，
  [VideoINSTA](https://arxiv.org/abs/2409.20365) / [GitHub](https://github.com/mayhugotong/VideoINSTA)。
  这类工作通常把增强放在 `shot / frame` 级采样、时间感知编码或信息充分性驱动的重看机制上，主要服务于**长视频理解能力**与**时序推理能力**的构建。

- 片段切分与语义一致 event 构建：
  [Panda-70M](https://arxiv.org/abs/2402.19479) / [GitHub](https://github.com/snap-research/Panda-70M)。
  这类工作更强调先把原始长视频变成语义一致、长度可控、便于后续 caption 和筛选的片段，因此其增强重点不是“如何写标注”，而是“如何先把可标注单元构造出来”。

- 时间结构化描述与 temporal grounding：
  [TimeChat-Captioner](https://arxiv.org/abs/2602.08711) / [GitHub](https://github.com/yaolinli/TimeChat-Captioner)，
  [TRACE](https://arxiv.org/abs/2410.05643) / [GitHub](https://github.com/gyxxyg/TRACE)。
  这类工作把增强放在 `time-aware captioning`、`causal event modeling` 和 `temporal grounding` 上，目标不是简单增加样本量，而是提升时间边界、事件结构和脚本化描述的质量。

- procedure / instructional 场景：
  [Paprika](https://arxiv.org/abs/2303.18230) / [GitHub](https://github.com/salesforce/paprika)，
  [SurgLLM](https://arxiv.org/abs/2509.00357) / [GitHub](https://github.com/franciszchen/SurgLLM)。
  这类工作服务的并不是开放域网页视频，而是 **instructional video** 或 **surgical procedure video**。它们更关注步骤结构、过程依赖、工具状态和领域术语，因此所需的数据增强策略也和通用网页视频明显不同。

- 多镜头叙事与审美 / 电影化视频：
  [Shot2Story](https://arxiv.org/abs/2312.10300) / [GitHub](https://github.com/bytedance/Shot2Story)，
  [CineTechBench](https://arxiv.org/abs/2505.15145) / [GitHub](https://github.com/PRIS-CV/CineTechBench)。
  这类工作更强调多镜头叙事连贯性、镜头语言、构图、运镜、光线和色彩等**电影化表达**，它们对应的数据增强重点已经不只是“看见什么”，而是“如何理解镜头组织与审美表达”。

- 对象中心与细粒度局部描述：
  [CAT-V](https://arxiv.org/abs/2504.05541) / [GitHub](https://github.com/yunlong10/CAT-V)。
  这类工作把增强放在对象级、区域级和时空联合的细粒度描述上，强调用户可指定目标、局部证据和精细 caption，而不是只生成全局概述。

因此，这些文献的专注领域并不一致：有些服务于**长视频理解能力**，有些更接近**模型推理与 grounding 能力**；同时它们面对的数据域也不同，有些是 procedure 类视频，有些是开放域网页视频，有些是多镜头叙事或电影化视频。正因为这些工作各自处在不同生态位、优化目标也不同，它们的数据增强技术很难直接横向比较，只能在不同的数据领域和不同需求下选择一条最优技术路径。

总而言之，我们最终没有把系统固定为某一篇文献的单一路线，而是把落脚点放在**高自由度、可复用、可组合**的视频算子系统上，使其能够支撑当前不同任务、不同领域、不同预算约束下的数据增强需求。


### 2.2 覆盖完整视频增强链路
首版 9 个算子必须覆盖从视频切分到最终标注与筛选的完整流程：

`segment -> clean -> sample -> dedup/compress -> text evidence -> spatial focus -> temporal summarize -> annotation -> quality gate`

### 2.3 强调数据流而不是单点技巧
算子系统的价值不在于某一个算子本身多么先进，而在于：

- 上游产物能否被下游稳定消费
- 中间态是否统一
- 数据血缘是否可追溯
- 同一段视频是否能在不同预算下被重用

### 2.4 与文献对齐
系统不追求把每一篇论文原封不动封装成一个黑盒，而是从 LVLM / Video-LLM 文献中提炼出稳定、可复用的原子能力，并明确哪些能力具有强文献支撑。

## 3. 9 个视频算子

| 编号 | 算子名称 | 核心职责 | 文献依赖等级 | 最新支撑强度 |
|---|---|---|---|---|
| A1 | 划分算子 | 把原视频切成时序一致的片段 | P1 论文拆解 | 中 |
| A2 | 片段清洗算子 | 过滤低质量、低信息密度片段 | P0 系统支撑 | 弱 |
| A3 | 采样算子 | 统一基础采样与自适应预算分配 | P2 论文原生 | 很强 |
| A4 | 去重算子 | 压缩冗余帧和冗余视觉表征 | P1 论文拆解 | 强 |
| A5 | 文本抽取算子 | 收集 OCR、字幕、ASR 等文本证据 | P1 论文拆解 | 中偏弱 |
| A6 | 空间聚焦算子 | 生成区域级视觉证据 | P2 论文原生 | 很强 |
| A7 | 时序定位与变化摘要算子 | 建模关键时刻和变化边界 | P2 论文原生 | 很强 |
| A8 | 标注生成算子 | 生成 caption、QA、summary 等标注 | P2 论文原生 | 很强 |
| A9 | 候选选择与质量裁决算子 | 选择优质候选并做最终过滤 | P2 论文原生 | 中强 |

这里的一个关键调整是：原 A4 自适应采样算子被并入 A3，作为 A3 的高阶模式存在。原因是两者输入输出完全一致，区别主要在采样策略，而不是数据流阶段。

文献依赖等级说明：

- `P2 论文原生`：该能力能直接映射到已有文献中的明确处理环节。
- `P1 论文拆解`：不是论文中的独立模块，但可稳定拆为原子算子。
- `P0 系统支撑`：主要服务于整条链路闭环，不主张作为某篇论文的直接复刻。

## 4. 系统分层

### 4.1 基础系统层
负责建立稳定的数据流入口和预算可控的基础处理能力。

- A1 划分算子
- A2 片段清洗算子
- A3 采样算子
- A4 去重算子
- A5 文本抽取算子

### 4.2 文献强驱动层
负责提供当前 Video-LLM 研究真正强调的高价值能力。

- A3 采样算子中的自适应模式
- A6 空间聚焦算子
- A7 时序定位与变化摘要算子
- A8 标注生成算子
- A9 候选选择与质量裁决算子

## 6. 系统设计约束

### 6.1 统一中间态
所有算子都必须通过同一份 `VideoPacket` 交互，不允许每个算子定义自己的私有协议。

### 6.2 追加式写入
算子只允许追加新对象或追加新状态，不允许覆写上游原始结果。

### 6.3 证据先于文本
任何最终标注都必须指向明确证据，不能让标注生成算子直接对原视频自由发挥。

### 6.4 模式与预算解耦
每个算子至少应该区分：

- 轻量模式：用于跑通链路和低成本批处理
- 平衡模式：用于通用网页混合视频
- 重模式：用于高质量样本生产

## 7. 当前版本的取舍
这套设计刻意没有把调度器、缓存、存储、分布式执行器纳入 9 个算子，因为这些属于系统基础设施而不是视频增强算子本身。

当前版本也没有把音频事件检测单独拆为新算子，而是把字幕、ASR、OCR 等统一归入 A5 文本抽取算子。这样可以在保证系统完整性的前提下控制复杂度，避免第一版过度膨胀。
