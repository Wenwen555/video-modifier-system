# 高自由度视频算子系统总览

## 1. 文档定位
本文件是 `算子系统/` 目录下的总设计文档，用于回答四个核心问题：

1. 这套系统为什么不是简单地堆更多工具，而是围绕一条标准数据流来组织。
2. 为什么当前版本仍然固定为 9 个视频算子，而不是继续无限扩张。
3. 默认主链路为什么收敛为 `A1 -> A2 -> A3 -> A7 -> A8 -> A4/A5/A6 -> A9`。
4. 哪些算子具有强文献锚点，哪些算子更多是把文献流程真正落地为可运行系统所必需的中间节点。

## 2. 设计目标

### 2.1 高自由度
这套系统同时支持两种使用方式：

- 原子化自由拼装：研究者可以按任务需要自由组合算子。
- 预设模板复用：可以快速复现若干论文风格或场景风格的处理链路。

采用此设定的原因，是因为当前 data-centric 文献并不在同一个位置做“视频数据增强”。有些工作把增强前置到 `shot / frame sampling`，目标是提升长视频理解中的关键信息保留；有些工作把增强前置到 `segment split + merge`，目标是先构造语义一致的 event 片段；还有一些工作把增强落在 `draft generation`、`temporal grounding`、`object-centric captioning` 或 `quality screening` 上。

换句话说，大家都在“做更好的视频数据”，但切入点并不相同。因此，系统必须既能表达一条当前默认主链路，也能保留 route-two 和 route-three 的旁路能力。

### 2.2 覆盖完整视频增强链路
当前落地版的 9 个算子必须覆盖从视频切分到 grounded 成品样本出站的完整流程。

默认主链路写成：

`segment -> context orchestrate -> sample -> draft -> query bundle -> temporal/spatial/text evidence -> grounded revision -> quality screening`

对应算子顺序写成：

`A1 -> A2 -> A3 -> A7 -> A8 -> A4/A5/A6 -> A9`

这里需要明确两点：

- 算子编号是系统内的稳定身份标识
- 不代表所有 preset 都必须严格按数字顺序串行执行

这也是为什么当前默认主链路中，A7 与 A8 会先于 A4/A5/A6 执行。

### 2.3 强调数据流而不是单点技巧
算子系统的价值不在于某一个算子本身多么先进，而在于：

- 上游产物能否被下游稳定消费
- 中间态是否统一
- 数据血缘是否可追溯
- 同一段视频是否能在不同预算下被重用

### 2.4 与文献对齐
系统不追求把每一篇论文原封不动封装成一个黑盒，而是从 LVLM / Video-LLM 文献中提炼出稳定、可复用的原子能力，并明确哪些能力具有强文献支撑。

## 3. 9 个视频算子

| 编号 | 算子名称 | 核心职责 | 文献依赖等级 | 最新支撑强度 |
|---|---|---|---|---|
| A1 | 划分算子 | 把原视频切成时序一致的片段 | P1 论文拆解 | 中 |
| A2 | 上下文编排算子 | 把离散片段整理成可消费的上下文单元 | P1 论文拆解 | 中 |
| A3 | 采样算子 | 统一基础采样与自适应预算分配 | P2 论文原生 | 很强 |
| A4 | 时序证据定位算子 | 按 query bundle 收缩时序范围并找出支持或反驳 claim 的关键 span | P2 论文原生 | 很强 |
| A5 | 空间证据聚焦算子 | 在候选 span 内生成对象级和区域级视觉证据 | P2 论文原生 | 很强 |
| A6 | 文本辅证抽取算子 | 在候选 span 内汇聚 OCR、字幕、ASR 等文本辅证 | P1 论文拆解 | 中偏弱 |
| A7 | 草案生成算子 | 生成粗 caption、QA 草案与 claim slots | P1 论文拆解 | 中强 |
| A8 | Query规范化算子 | 把草案或外部问题改写成 grounding query bundles，并控制预算路由 | P1 论文拆解 | 中 |
| A9 | Grounded回修与质量筛查算子 | 证据摘要、答案回修、严格审查与最终筛查 | P2 论文原生 | 很强 |

这里的一个关键保留是：系统依然固定为 9 个原子算子，但把后半段语义从旧版的“问题规划 -> QA生成 -> 核验回修”重写为“草案生成 -> Query规范化 -> Grounded回修与质量筛查”。

这样做的原因是：

- 这更贴近你当前确定的路线一默认主链路
- 它仍能兼容路线二中 `external question -> A8 -> A4/A5/A6 -> A9` 的推理旁路
- 它避免把 `证据摘要 + 答案回修 + 严格审查 + 质量筛查` 误拆成站点上尚未稳定的新算子数量

文献依赖等级说明：

- `P2 论文原生`：该能力能直接映射到已有文献中的明确处理环节。
- `P1 论文拆解`：不是论文中的独立模块，但可稳定拆为原子算子。
- `P0 系统支撑`：主要服务于整条链路闭环，不主张作为某篇论文的直接复刻。

## 4. 系统分层

### 4.1 基础系统层
负责建立稳定的数据流入口和预算可控的基础处理能力。

- A1 划分算子
- A2 上下文编排算子
- A3 采样算子

### 4.2 草案与 Query 层
负责先形成低成本草案，再把草案改写成可执行的 grounding query。

- A7 草案生成算子
- A8 Query规范化算子

### 4.3 Grounding 回看层
负责围绕 query bundle 回到视频中取证。

- A4 时序证据定位算子
- A5 空间证据聚焦算子
- A6 文本辅证抽取算子

### 4.4 终审与出站层
负责 grounded 回修、严格审查和最终筛查。

- A9 Grounded回修与质量筛查算子

## 6. 系统设计约束

### 6.1 统一中间态
所有算子都必须通过同一份 `VideoPacket` 交互，不允许每个算子定义自己的私有协议。

### 6.2 追加式写入
算子只允许追加新对象或追加新状态，不允许覆写上游原始结果。

### 6.3 最终结果必须证据闭环
A7 允许先生成草案文本，但任何最终接受的 QA 都必须经过 `A8 -> A4/A5/A6 -> A9` 的 grounded 闭环，不能让草案直接作为最终样本出站。

### 6.4 编号稳定但顺序可变
算子编号是站点、协议和实现中的稳定标识；实际执行顺序由 preset 和 route 决定，而不是被数字顺序锁死。

### 6.5 模式与预算解耦
每个算子至少应该区分：

- 轻量模式：用于跑通链路和低成本批处理
- 平衡模式：用于通用网页混合视频
- 重模式：用于高质量样本生产

## 7. 当前版本的取舍
这套设计刻意没有把调度器、缓存、存储、分布式执行器纳入 9 个算子，因为这些属于系统基础设施而不是视频增强算子本身。

当前版本也没有把“严格审查 Agent”单独拆为第 10 个算子，而是把它收进 A9 的内部阶段。这样做不是否认它的重要性，而是为了让第一版系统保持 9 算子的稳定外壳，同时把最关键的落地点先跑通。
