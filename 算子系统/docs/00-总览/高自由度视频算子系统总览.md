# 高自由度视频算子系统总览

## 1. 文档定位
本文件是 `算子系统/` 目录下的总设计文档，用于回答五个核心问题：

1. 这套系统为什么面向 `post-training / instruction tuning`，而不是在线问答推理。
2. 这套系统为什么不再以“后验 grounding 回看链”为默认路线，而改成“结构骨架优先”的数据构造链。
3. 为什么当前版本固定为 10 个视频算子。
4. 哪些算子对应文献中的强模式，哪些算子是将文献流程系统化落地所必需的中间节点。
5. 这条链路为什么更适合当前强调的高质量 LVLM QA 数据集构造任务。

## 2. 系统定位

### 2.1 面向 post-training / instruction tuning
当前系统的默认目标不是做一次性的 benchmark 推理，也不是做 question-given 的在线回看，而是为 LVLM 的 `post-training / instruction tuning` 自动构造高质量样本。

这里的“高质量”至少包含四层含义：

- 样本要能形成可训练的 `Question / Answer / Evidence`
- 样本要带有可回挂的结构骨架与证据血缘
- 样本要能被回修、筛查、分层和复用
- 样本要能区分普通 QA、grounded QA 和强 grounding QA

因此，系统默认关注的是“数据出站协议”，而不是“单次回答得对不对”。

### 2.2 面向数据构造而不是单次推理
当前系统的默认落脚点不是在线问答，而是：

- 自动构造 `Question / Answer / Evidence`
- 保留可追溯的结构骨架
- 让最终样本可以被筛查、回修和复用

因此，系统默认路线不再是“先问题，再回视频找答案”，而是：

`视频 -> 结构骨架 / 证据单元 -> Q/A/E 物化 -> 回修 -> 质量筛查`

### 2.3 默认主链仍然是 structure-first
当前落地版的 10 个算子覆盖如下默认主链路：

`A1 划分 -> A2 上下文编排 -> A3 采样 -> A4 多粒度描述 -> A5 时序结构抽取 -> A6 文本辅证抽取 -> A7 结构骨架构建 -> A8 Q/A/E 物化 -> A9 一致性回修 -> A10 质量筛查`

这条链路对应的高层流程是：

`segment -> light orchestration -> sample -> multi-granular description -> temporal/text/evidence normalization -> structural scaffold -> triplet materialization -> revision -> screening`

需要特别说明的是，A2 在当前版本中被明确收紧为“轻量入口层”：

- 只保留 `MERGE_STITCH`、`WINDOW_ATTACH`、`FILTER_HINT`
- 不承担语义 pack、事件层级或骨架搭建
- `hierarchical pack` 明确后移到 A7

### 2.4 为什么要兼容三条数据主线
当前视频 QA / grounded QA 数据构造并不只有一种做法。对 post-training 数据集而言，最常见、也最值得兼容的是三条主线：

- `narration / caption-first`
- `structure-first`
- `QA-first / post-hoc grounding`

你们当前系统已经天然适配第二条，但如果只把 A7/A8 理解成“structure-first 专用算子”，那么第一条和第三条都会退化成文档里的“旁路”，无法真正落到统一协议里。

因此，当前版本的演进方向不是新增第 11 个算子，而是把：

- `A7` 从“只接结构构件”扩展为“统一收敛层”
- `A8` 从“只做 joint synthesis”扩展为“最终 triplet 物化层”

### 2.5 多入口、统一出口
系统现在采用的不是“一个主链 + 两个例外”，而是：

- 上游允许不同来源进入：video-derived structure、narration chunks、external QA drafts
- 中游统一规约为：`evidence_units`、`scaffold_units`、`qa_drafts`
- 下游统一出站为：`qae_triplets -> revisions -> quality_records`

这意味着系统真正统一的，不是“问题从哪来”，而是：

- 证据如何表示
- triplet 如何定稿
- 数据如何被筛查成可用于 instruction tuning 的高质量样本

## 3. 10 个视频算子

| 编号 | 算子名称 | 核心职责 | 文献依赖等级 | 最新支撑强度 |
|---|---|---|---|---|
| A1 | 划分算子 | 把原视频切成时序一致的片段 | P1 论文拆解 | 中 |
| A2 | 上下文编排算子 | 只执行 `MERGE_STITCH`、`WINDOW_ATTACH`、`FILTER_HINT` | P1 论文拆解 | 中 |
| A3 | 采样算子 | 统一基础采样与预算感知采样 | P2 论文原生 | 很强 |
| A4 | 多粒度描述算子 | 生成 frame/event/segment/context 等多层级描述，为证据规约和骨架构建提供语义底座 | P1 论文拆解 | 中 |
| A5 | 时序结构抽取算子 | 抽取事件、时间窗、阶段关系和多跳时序线索 | P2 论文原生 | 很强 |
| A6 | 文本辅证抽取算子 | 汇聚 OCR、字幕、ASR 等文本信号 | P1 论文拆解 | 中 |
| A7 | 结构骨架构建算子 | 把描述、时序结构、文本辅证、证据单元或 QA 草案组织成 chunk、pack、tree、graph 或 query scaffold | P1 论文拆解 | 中强 |
| A8 | Q/A/E共同合成算子 | 在骨架或草案约束下联合生成或装配 Question、Answer 和 Evidence，并产出最终 triplet | P2 论文原生 | 很强 |
| A9 | 一致性回修算子 | 对 Q/A/E 三元组做字段对齐、去冲突和局部修订 | P1 论文拆解 | 中强 |
| A10 | 质量筛查算子 | 依据规则、分数和训练价值做最终筛查，输出适合 post-training 的样本子集 | P2 论文原生 | 中强 |

## 4. 路由家族 -> 算子分工总表

| 路由家族 | 典型上游输入 | A7 的角色 | A8 的角色 | 最终适合的数据 |
|---|---|---|---|---|
| `narration / caption-first` | narration chunks、ASR、字幕、dense captions | 组织 narration scaffold 或 caption-event scaffold | 共同生成 `Q/A/E` | 大规模 QA、弱监督 grounded QA |
| `structure-first` | temporal structures、event tree、action graph、hierarchical tree | 构建可复用结构骨架 | 从骨架直接物化 triplets | 高质量 grounded QA、强证据 QA |
| `QA-first / post-hoc grounding` | 人工 QA、外部 benchmark QA、LM draft QA | 围绕 question 构造 query scaffold / verification scaffold | 绑定证据、校验或补全 answer，生成最终 triplets | 人工高质 grounded QA、benchmark 升级数据 |

这张表的关键含义是：

- A4/A5/A6 是“证据与结构构件层”
- A7 是三条数据主线的统一收敛层
- A8 是三条主线共享的 triplet 物化层
- A9/A10 负责把候选样本真正变成可用于 LVLM post-training 的高质量数据

## 5. 为什么仍然是这 10 个

### 5.1 不单列“外部 QA 导入算子”
这是一个刻意取舍。

原因是：

- 外部 question / answer / benchmark 标注更像数据接入方式，而不是视频构造算子
- 真正需要统一的是它们进入系统后的规约方式
- 因此我们把这部分能力落在 `VideoPacket` 的 `qa_drafts[]` 与 `grounding_records[]` 协议上，而不是新增一个计数算子

### 5.2 为什么要把“结构骨架构建”和“Q/A/E 物化”拆开
如果把两者混在一起：

- 结构可复用性差
- narration-first 和 QA-first 路线难以共享中间层
- 不便于后续重采样、重生题和人工回修

因此当前系统把：

- A7：构建或绑定 scaffold
- A8：利用 scaffold 物化最终 Q/A/E

明确拆开。

### 5.3 为什么要单列 A9 和 A10
A8 输出的只是候选 triplets，不应直接出站。

对于 post-training / instruction tuning，真正需要的是：

- 一致性稳定
- grounding 完整
- 训练价值明确

因此需要：

- A9 做一致性回修
- A10 做最终筛查和数据分层

## 8. 系统设计约束

### 8.1 统一中间态
所有算子都必须通过同一份 `VideoPacket` 交互。

### 8.2 追加式写入
算子只允许追加新对象和新状态，不允许覆写上游原始结果。

### 8.3 triplet 物化必须依赖规约后的中间层
A8 不允许绕过 A7 或 `qa_drafts[] / evidence_units[]` 直接对原视频自由生成 Q/A/E。

### 8.4 最终样本必须经过回修与筛查
A8 输出的是候选样本，不是最终数据；必须经过 A9 和 A10 才能出站。

### 8.5 默认目标是高质量 LVLM post-training 数据
系统当前默认优化目标不是“尽量多生题”，而是“生成更可靠、更可控、更适合作为 instruction tuning 监督信号的数据”。

## 9. 当前版本的取舍
这套设计刻意没有把调度器、缓存、分布式执行器纳入 10 个算子，因为这些属于基础设施而不是视频数据构造算子本身。

当前版本也没有把“人工审核”单独拆为第 11 个默认算子，而是把它视为 A10 之后的可选出口。当前的重点是先把：

- 三条主要数据主线纳入统一协议
- 高质量 `Q/A/E` 出站链打通
- 面向 LVLM post-training 的数据质量控制明确化
