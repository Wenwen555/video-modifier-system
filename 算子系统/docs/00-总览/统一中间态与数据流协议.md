# 统一中间态与数据流协议

## 1. 设计目标
统一中间态的目的是让所有算子都遵循同一套输入输出语言，避免出现以下问题：

- 每个算子定义自己的 JSON 结构，导致下游难以复用
- 同一视频在不同链路中反复解码和重复计算
- 最终数据无法回溯到具体片段、采样点和骨架节点
- Q/A/E 三者脱钩，无法解释它们为何属于同一条样本

系统统一使用 `VideoPacket` 作为唯一中间态容器。

## 2. 顶层结构

```json
{
  "video": {},
  "trace": {},
  "segments": [],
  "samples": [],
  "coarse_captions": [],
  "temporal_structures": [],
  "text_signals": [],
  "scaffold_units": [],
  "qae_triplets": [],
  "revisions": [],
  "quality_records": []
}
```

## 3. 顶层字段说明

### 3.1 video
记录原视频和基础元数据。

### 3.2 trace
记录整条链路的追踪信息，如 `job_id`、`pipeline_name`、`budget_profile`。

### 3.3 segments
由 A1 和 A2 共同维护，表示所有可消费片段及其上下文状态。

### 3.4 samples
由 A3 维护，表示代表帧和采样结果。

### 3.5 coarse_captions
由 A4 维护，表示片段级或上下文级粗描述。

### 3.6 temporal_structures
由 A5 维护，表示时间窗、事件链、阶段关系和多跳时序线索。

### 3.7 text_signals
由 A6 维护，表示 OCR、字幕、ASR 和其他文本辅证。

### 3.8 scaffold_units
由 A7 维护，表示 chunk、tree node、graph node 或其他结构骨架单元。

### 3.9 qae_triplets
由 A8 维护，表示共同合成得到的 `Question / Answer / Evidence` 三元组。

### 3.10 revisions
由 A9 维护，表示针对 triplet 的一致性回修记录。

### 3.11 quality_records
由 A10 维护，表示筛查分数、保留决策和拒绝原因。

## 4. 公共对象元信息
所有数组中的对象都必须包含以下公共字段：

- `id`
- `source_ids`
- `operator_name`
- `operator_version`
- `confidence`
- `status`
- `resource_refs`

状态字段推荐枚举：

- `active`
- `filtered`
- `superseded`
- `rejected`
- `accepted`

## 6. 关键对象的补充约束

### 6.1 ScaffoldUnit 必须是可复用结构
`scaffold_units[]` 不允许只是把上游字段简单拼接成一个长字符串。

每个 `scaffold_unit` 至少还应显式说明：

- `unit_type`
- `scaffold_summary`
- `composition_rule`
- `child_refs`
- `evidence_distribution`

### 6.2 QAETriplet 必须显式绑定证据
每条 `qae_triplet` 至少应引用以下证据之一：

- `temporal_unit_id`
- `caption_id`
- `text_signal_id`
- `segment_id`

如果是多跳问题，则应显式保留多个证据引用。

## 7. 算子调用契约
每个算子统一消费如下三类输入：

- `VideoPacket`
- `OperatorConfig`
- `Selector`

### 7.1 OperatorConfig
定义算子的运行模式和参数。

推荐字段：

- `mode`
- `budget_profile`
- `thresholds`
- `outputs_required`
- `retry_policy`

### 7.2 Selector
定义本次算子处理哪些对象。

推荐字段：

- `segment_ids`
- `sample_ids`
- `caption_ids`
- `temporal_unit_ids`
- `scaffold_ids`
- `triplet_ids`
- `only_active`

## 8. 追加式写入规则
为保证链路可回溯，所有算子都要遵守以下规则：

1. 不覆写上游原始对象。
2. 需要替代旧结果时，新增对象并把旧对象状态设为 `superseded`。
3. 过滤行为必须显式写入 `filtered` 或 `rejected`。
4. 最终保留结果必须显式写入 `accepted`。

## 9. 数据流规则

### 9.1 固定主干
`video -> segments -> samples -> coarse_captions / temporal_structures / text_signals -> scaffold_units -> qae_triplets -> revisions -> quality_records`

### 9.2 共同合成禁止直连原视频
A8 QAE 共同合成算子不允许绕过 `scaffold_units` 直接读取原视频自由生成三元组。

### 9.3 筛查前必须回修
A10 不应直接消费 A8 的原始 triplets，而应优先消费经过 A9 回修后的版本。

## 10. 最小示例

```json
{
  "video": {
    "video_id": "vid_001",
    "video_uri": "s3://bucket/raw/vid_001.mp4",
    "metadata": {
      "fps": 30,
      "total_frames": 3600,
      "duration_sec": 120.0
    }
  },
  "trace": {
    "job_id": "job_20260408_001",
    "pipeline_name": "scaffold_first_default",
    "budget_profile": "balanced"
  },
  "segments": [],
  "samples": [],
  "coarse_captions": [],
  "temporal_structures": [],
  "text_signals": [],
  "scaffold_units": [],
  "qae_triplets": [],
  "revisions": [],
  "quality_records": []
}
```

## 11. 原则

- 统一协议优先于单算子优化
- 结构骨架优先于自由生成
- 共同合成优先于后验拼接
- 最终接受样本必须可回挂到结构证据
