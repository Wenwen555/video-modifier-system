# 统一中间态与数据流协议

## 1. 设计目标
统一中间态的目的是让所有算子都遵循同一套输入输出语言，避免出现以下问题：

- 每个算子都定义自己的 JSON 结构，导致下游难以复用
- 同一视频在不同链路中反复解码和重复计算
- 最终标注无法回溯到具体片段、帧和局部证据
- 草案、query、证据和最终 QA 之间没有显式对应关系

系统统一使用 `VideoPacket` 作为唯一中间态容器。

## 2. 顶层结构
`VideoPacket` 顶层字段固定如下：

```json
{
  "video": {},
  "trace": {},
  "segments": [],
  "samples": [],
  "draft_units": [],
  "query_bundles": [],
  "text_signals": [],
  "regions": [],
  "temporal_events": [],
  "qa_pairs": [],
  "verification": []
}
```

## 3. 顶层字段说明

### 3.1 video
记录原视频和基础元数据。

推荐字段：

- `video_id`
- `video_uri`
- `metadata.fps`
- `metadata.total_frames`
- `metadata.duration_sec`
- `metadata.resolution`
- `subtitle_uri`
- `asr_uri`

### 3.2 trace
记录整条数据链路的追踪信息。

推荐字段：

- `job_id`
- `created_at`
- `pipeline_name`
- `budget_profile`
- `operator_history`

### 3.3 segments
由 A1 和 A2 共同维护，表示所有可消费片段及其上下文状态。

### 3.4 samples
由 A3 维护，表示代表帧和采样结果。

### 3.5 draft_units
由 A7 维护，表示粗 caption、QA 草案、claim slots 和答案约束。

### 3.6 query_bundles
由 A8 维护，表示规范化后的 temporal / spatial / textual grounding query 与预算路由策略。

### 3.7 text_signals
由 A6 维护，表示 OCR、字幕、ASR 和文本融合结果。

### 3.8 regions
由 A5 维护，表示局部框、mask、crop 或 SoM 标记。

### 3.9 temporal_events
由 A4 维护，表示时间边界、关键 span、状态变化和显式时序证据。

### 3.10 qa_pairs
由 A9 维护，表示 grounded 回修后的最终问答候选。

### 3.11 verification
由 A9 维护，表示严格审查、质量筛查和最终保留决策。

## 4. 公共对象元信息
所有数组中的对象都必须包含以下公共字段：

- `id`
- `source_ids`
- `operator_name`
- `operator_version`
- `confidence`
- `status`
- `resource_refs`

状态字段推荐枚举：

- `active`
- `filtered`
- `superseded`
- `rejected`
- `accepted`

## 6. 关键对象的补充约束

### 6.1 DraftUnit 的最小结构化要求
为保证路线一可稳定落地，`draft_units[]` 不允许只保留纯自然语言问答草案。

每条 `draft_unit` 至少还应补充：

- `claim_slots[].claim_id`
- `claim_slots[].claim_text`
- `claim_slots[].answer_role`
- `claim_slots[].evidence_required`
- `claim_slots[].query_seeds`
- `uncertainty_points`
- `budget_hint`
- `rejection_policy`

### 6.2 QueryBundle 的最小路由要求
每条 `query_bundle` 至少应显式说明：

- 是否运行 `A4`
- 是否运行 `A5`
- 是否运行 `A6`
- 哪些 claim 是 `must_support_claim_ids`
- 当前是 `draft-conditioned` 还是 `external-question-conditioned`

## 7. 算子调用契约
每个算子统一消费如下三类输入：

- `VideoPacket`
- `OperatorConfig`
- `Selector`

### 7.1 OperatorConfig
定义算子的运行模式和参数。

推荐字段：

- `mode`
- `budget_profile`
- `thresholds`
- `outputs_required`
- `retry_policy`

### 7.2 Selector
定义本次算子处理哪些对象。

推荐字段：

- `segment_ids`
- `sample_ids`
- `draft_ids`
- `bundle_ids`
- `qa_ids`
- `only_active`

## 8. 追加式写入规则
为保证链路可回溯，所有算子都要遵守以下规则：

1. 不覆写上游原始对象。
2. 需要替代旧结果时，新增对象并把旧对象状态设为 `superseded`。
3. 过滤行为必须显式写入 `filtered` 或 `rejected`。
4. 最终保留结果必须显式写入 `accepted`。

## 9. 数据流规则

### 9.1 固定主干
`video -> segments -> samples -> draft_units -> query_bundles -> evidence -> qa_pairs -> verification`

### 9.2 证据引用
每一条 `qa_pair` 至少要引用以下证据之一：

- `event_id`
- `region_id`
- `text_signal_id`
- `sample_id`
- `segment_id`

### 9.3 草案不等于最终结果
A7 允许生成草案文本，但任何 `qa_pairs[].status=accepted` 的最终结果都不能绕过 `query_bundles + explicit evidence + verification` 直接出站，否则会破坏 grounded 可解释性。

### 9.4 路线二兼容规则
当系统走 route-two 时，`query_bundle.source_draft_id` 可以为空，但必须额外保留：

- `external_question_id`
- `conditioning_type=external_question`

## 11. 原则

- 统一协议优先于单算子优化
- 可复用优先于一次性 pipeline
- claim-level 可核验优先于整句流畅
- 最终接受样本必须能回挂到显式证据
