# 统一中间态与数据流协议

## 1. 设计目标
统一中间态的目的是让所有算子都遵循同一套输入输出语言，避免出现以下问题：

- 每个算子都定义自己的 JSON 结构，导致下游难以复用
- 同一视频在不同链路中反复解码和重复计算
- 最终标注无法回溯到具体片段、帧和局部证据

系统统一使用 `VideoPacket` 作为唯一中间态容器。

## 2. 顶层结构
`VideoPacket` 顶层字段固定如下：

```json
{
  "video": {},
  "trace": {},
  "segments": [],
  "samples": [],
  "text_signals": [],
  "regions": [],
  "temporal_events": [],
  "question_plans": [],
  "qa_pairs": [],
  "verification": []
}
```

## 3. 顶层字段说明

### 3.1 video
记录原视频和基础元数据。

推荐字段：

- `video_id`
- `video_uri`
- `metadata.fps`
- `metadata.total_frames`
- `metadata.duration_sec`
- `metadata.resolution`
- `subtitle_uri`
- `asr_uri`

### 3.2 trace
记录整条数据链路的追踪信息。

推荐字段：

- `job_id`
- `created_at`
- `pipeline_name`
- `budget_profile`
- `operator_history`

### 3.3 segments
由 A1 和 A2 共同维护，表示所有可消费片段及其上下文状态。

### 3.4 samples
由 A3 维护，表示代表帧和采样结果。

### 3.5 text_signals
由 A6 维护，表示 OCR、字幕、ASR 和文本融合结果。

### 3.6 regions
由 A5 维护，表示局部框、mask、crop 或 SoM 标记。

### 3.7 temporal_events
由 A4 维护，表示时间边界、状态变化和时序摘要。

### 3.8 question_plans
由 A7 维护，表示题型、目标证据、答案形式和难度等规划结果。

### 3.9 qa_pairs
由 A8 维护，表示最终生成的问答候选。

### 3.10 verification
由 A9 维护，表示 grounded 核验、回修和最终保留决策。

## 4. 公共对象元信息
所有数组中的对象都必须包含以下公共字段：

- `id`
- `source_ids`
- `operator_name`
- `operator_version`
- `confidence`
- `status`
- `resource_refs`

状态字段推荐枚举：

- `active`
- `filtered`
- `superseded`
- `rejected`
- `accepted`

## 5. 各类对象的最小字段

### 5.1 Segment
- `segment_id`
- `timestamp_range`
- `frame_range`
- `duration_sec`
- `confidence`

### 5.2 Sample
- `sample_id`
- `segment_id`
- `timestamp_sec`
- `frame_index`
- `sample_type`

### 5.3 TextSignal
- `text_signal_id`
- `segment_id`
- `timestamp_range`
- `text_type`
- `text`

### 5.4 Region
- `region_id`
- `sample_id`
- `region_type`
- `bbox` 或 `mask_uri`

### 5.5 TemporalEvent
- `event_id`
- `segment_id`
- `start_sec`
- `end_sec`
- `event_type`
- `summary`

### 5.6 QuestionPlan
- `plan_id`
- `question_intent`
- `evidence_refs`
- `answer_type`
- `difficulty`

### 5.7 QAPair
- `qa_id`
- `question`
- `answer`
- `plan_id`
- `evidence_refs`
- `status`

### 5.8 VerificationRecord
- `verification_id`
- `target_qa_id`
- `scores`
- `decision`
- `reason`

## 6. 算子调用契约
每个算子统一消费如下三类输入：

- `VideoPacket`
- `OperatorConfig`
- `Selector`

### 6.1 OperatorConfig
定义算子的运行模式和参数。

推荐字段：

- `mode`
- `budget_profile`
- `thresholds`
- `outputs_required`
- `retry_policy`

### 6.2 Selector
定义本次算子处理哪些对象。

推荐字段：

- `segment_ids`
- `sample_ids`
- `plan_ids`
- `qa_ids`
- `only_active`

## 7. 追加式写入规则
为保证链路可回溯，所有算子都要遵守以下规则：

1. 不覆写上游原始对象。
2. 需要替代旧结果时，新增对象并把旧对象状态设为 `superseded`。
3. 过滤行为必须显式写入 `filtered` 或 `rejected`。
4. 最终保留结果必须显式写入 `accepted`。

## 8. 数据流规则

### 8.1 固定主干
`video -> segments -> samples -> evidence -> question_plans -> qa_pairs -> verification`

### 8.2 证据引用
每一条 `qa_pair` 至少要引用以下证据之一：

- `segment_id`
- `sample_id`
- `text_signal_id`
- `region_id`
- `event_id`

### 8.3 禁止直连
A8 QA生成算子不允许绕过中间态直接读取原视频并自由生成文本，否则会破坏可解释性。

## 9. 最小示例
```json
{
  "video": {
    "video_id": "vid_001",
    "video_uri": "s3://bucket/raw/vid_001.mp4",
    "metadata": {
      "fps": 30,
      "total_frames": 3600,
      "duration_sec": 120.0
    }
  },
  "trace": {
    "job_id": "job_20260330_001",
    "pipeline_name": "balanced",
    "budget_profile": "balanced"
  },
  "segments": [],
  "samples": [],
  "text_signals": [],
  "regions": [],
  "temporal_events": [],
  "question_plans": [],
  "qa_pairs": [],
  "verification": []
}
```

## 10. 原则
- 统一协议优先于单算子优化
- 可复用优先于一次性 pipeline
