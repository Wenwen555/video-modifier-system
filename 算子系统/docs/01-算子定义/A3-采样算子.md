# A3-采样算子

## 1. 目标
A3 负责在 A1 输出的片段上选取少量代表帧，作为后续文本抽取、空间聚焦、变化摘要和标注生成的统一输入。

当前参考实现位于：

- `01-算子定义/impl/a3_sampling_operator.py`

实现类为：

- `A3_Sampling_Operator`

基类为：

- `Modifier`

这意味着它与 A1 一样，既可以通过 `process(instance)` 处理单条样本，也可以通过 `run(dataset)` 批量处理数据集。

## 2. 当前实现范围
当前代码实现的是一版轻量、metadata-first 的采样算子，核心目标是统一采样协议，而不是直接完成帧解码、帧落盘或复杂质量判断。

当前版本支持：

- `CENTER`
- `UNIFORM`
- `HEAD_TAIL`
- `STATE`

当前版本不包含：

- 真实图像帧导出
- 模糊帧检测
- 运动感知采样
- 语义驱动采样
- 多模态打分重排

## 3. Mode 对照

| Mode | 采样依据 | 必需附加输入 | 关键参数 | 输出附加字段 | 适用场景 |
|---|---|---|---|---|---|
| `CENTER` | 片段中心帧 | 无 | 无 | `sample_type=CENTER` | 稳定镜头、快速概览、轻量基线 |
| `UNIFORM` | 在片段内部做均匀采样 | 无 | `num_frames` `target_fps` `max_frames_per_segment` | `sample_type=UNIFORM` | 长片段概览、作为所有高级采样的基线 |
| `HEAD_TAIL` | 提取首帧和尾帧 | 无 | `max_frames_per_segment` | `sample_type=HEAD/TAIL` | 状态对比、前后变化判断 |
| `STATE` | 在首尾附近做偏移采样，避开瞬时边界帧 | 无 | `padding_frames` `max_frames_per_segment` | `sample_type=STATE_HEAD/STATE_TAIL` | 关注状态变化，但希望避开切点附近不稳定帧 |

## 4. 算法行为

### 4.1 CENTER
`CENTER` 模式对每个片段只采 1 帧。

实现方式：

- 取 `frame_range.start_frame` 和 `frame_range.end_frame - 1` 之间的中点帧
- 返回单个样本

该模式是当前最稳定、成本最低的默认采样模式。

### 4.2 UNIFORM
`UNIFORM` 模式在片段内做均匀采样。

实现方式：

1. 读取片段起止帧。
2. 计算可用帧数 `available_frames`。
3. 若设置了 `target_fps`，则采样帧数为 `ceil(duration_sec * target_fps)`。
4. 否则采样帧数使用 `num_frames`。
5. 采样帧数至少为 `1`，且不会超过片段内可用帧数。
6. 在起止帧之间按等间隔生成采样帧索引。

如果最终启用了 `max_frames_per_segment`，则会在生成索引后再做一次截断。

### 4.3 HEAD_TAIL
`HEAD_TAIL` 模式每个片段采 2 帧：

- 首帧：`start_frame`
- 尾帧：`end_frame - 1`

输出中的 `sample_type` 分别为：

- `HEAD`
- `TAIL`

该模式适合关注片段前后状态是否变化。

### 4.4 STATE
`STATE` 模式是在 `HEAD_TAIL` 的基础上加入首尾偏移。

实现方式：

- 偏移量由 `padding_frames` 控制
- 实际偏移量不会超过片段长度的一半
- 头部采样位置为 `start_frame + offset`
- 尾部采样位置为 `last_frame - offset`

输出中的 `sample_type` 分别为：

- `STATE_HEAD`
- `STATE_TAIL`

该模式的目的不是增加采样数量，而是尽量避开切点附近可能存在的黑帧、模糊帧或过渡帧。

## 5. 输入协议
A3 的 `process(instance)` 接受单条字典样本。当前实现主要依赖 A1 输出的 `segments`，并在原记录上追加 `samples` 和 `sampling_result`。

### 5.1 必需字段

| 字段 | 说明 |
|---|---|
| `video_path` 或 `video_uri` | 视频路径。`video_path` 来自 `Modifier.video_key`，也兼容直接传入 `video_uri`。 |
| `segments` | A1 产出的片段列表。 |
| `segments[].frame_range.start_frame` | 片段起始帧。 |
| `segments[].frame_range.end_frame` | 片段结束帧。 |

### 5.2 推荐字段

| 字段 | 说明 |
|---|---|
| `video_id` | 视频唯一标识。缺失时自动由文件名生成。 |
| `video_metadata.fps` 或 `fps` | 帧率。 |
| `video_metadata.total_frames` 或 `total_frames` | 总帧数。 |
| `video_metadata.duration_sec` 或 `duration_sec` | 视频总时长。 |
| `segments[].segment_id` | 片段 ID。缺失时自动生成。 |

### 5.3 片段归一化逻辑
当前实现对输入片段有一个非常明确的要求：必须提供 A1 风格的 `frame_range`。

归一化时会做以下处理：

- 限制 `start_frame` 不小于 `0`
- 限制 `end_frame` 不超过 `video.total_frames`
- 确保 `end_frame >= start_frame + 1`
- 根据帧索引和 `fps` 回推 `start_sec`、`end_sec`
- 若缺少 `segment_id`，则自动生成 `seg_0001` 这类编号

当前实现不会从 `timestamp_range` 反推采样边界，它直接以 `frame_range` 作为唯一采样基准。

### 5.4 视频元数据补全逻辑
如果 `fps` 缺失，或者 `total_frames` 与 `duration_sec` 同时缺失，算子会尝试使用 OpenCV 自动探测视频元数据。

如果探测失败，会直接报错。

## 6. 输出协议
`process(instance)` 会在原记录上追加以下字段：

| 字段 | 说明 |
|---|---|
| `video_id` | 补全后的视频 ID。 |
| `video_uri` | 归一化后的视频路径。 |
| `video_metadata` | 标准化后的视频元数据。 |
| `samples` | 最终采样结果列表。 |
| `sampling_result` | 完整算子输出，包括全局统计。 |
| `sampling_mode` | 实际执行的采样模式。 |
| `sampling_metrics` | 当前包含 `total_segments`、`total_samples`、`processing_time_ms`。 |

每个 `sample` 至少包含：

| 字段 | 说明 |
|---|---|
| `id` / `sample_id` | 样本 ID，格式为 `smp_000001`。 |
| `segment_id` | 所属片段 ID。 |
| `source_ids` | 来源 ID 列表，当前包含视频 ID 和片段 ID。 |
| `timestamp_sec` | 采样时间戳。 |
| `frame_index` | 采样帧索引。 |
| `sample_type` | 当前样本类型。 |
| `sampling_mode` | 当前采样模式。 |
| `relative_position` | 样本在片段内部的相对位置，范围约为 `[0, 1]`。 |
| `sample_rank` | 该样本在片段内的排序编号。 |
| `samples_in_segment` | 当前片段的样本总数。 |
| `is_representative` | 当前固定为 `True`。 |
| `confidence` | 当前固定为 `1.0`。 |
| `status` | 当前固定为 `active`。 |
| `resource_refs.video_uri` | 当前仅保留源视频路径引用。 |
| `operator_name` | 当前算子名。 |
| `operator_version` | 当前实现版本。 |

需要注意的是，当前实现输出的是采样决策元数据，不直接输出 `frame_uri` 或实际图像文件。

## 7. 参数说明

| 参数 | 默认值 | 作用 |
|---|---:|---|
| `sampling_mode` | `CENTER` | 选择采样模式。 |
| `num_frames` | `4` | `UNIFORM` 模式默认采样帧数。 |
| `padding_frames` | `2` | `STATE` 模式的首尾偏移帧数。 |
| `target_fps` | `None` | `UNIFORM` 模式下按目标帧率决定采样密度。 |
| `max_frames_per_segment` | `None` | 对单个片段的输出样本数做硬上限限制。 |
| `strict` | `True` | 异常是否直接抛出。 |

## 8. 后处理规则
当前实现内置了两层轻量后处理：

### 8.1 去重保持顺序
如果 mode 产生了重复帧索引，算子会：

- 去重
- 保持原顺序不变

这在极短片段上尤其重要，可以避免 `HEAD_TAIL` 或 `STATE` 输出重复样本。

### 8.2 单片段样本数限制
如果设置了 `max_frames_per_segment`，算子会在 mode 选出候选帧之后再截断输出数量。

这意味着：

- mode 决定“候选采样位置”
- `max_frames_per_segment` 决定“最多保留多少个样本”

## 9. 错误处理
当前实现支持两种行为：

| `strict` | 行为 |
|---|---|
| `True` | 直接抛出异常，适合离线构建、调试和质量检查。 |
| `False` | 返回空 `samples`，并在 `sampling_result` 与 `sampling_error` 中写入错误信息。 |

常见报错来源包括：

- 缺少 `video_path` 或 `video_uri`
- 缺少 `segments`
- `segments[].frame_range` 不是合法字典
- 缺少 `frame_range.start_frame` 或 `frame_range.end_frame`
- `sampling_mode` 不受支持
- `target_fps <= 0`
- 缺少视频元数据且无法自动探测
- 视频文件无法打开

## 10. 与 `op_util.py` 的对齐方式
当前实现遵循 `Modifier` 的使用习惯，适合直接接入现有算子系统。

对齐点如下：

- 通过 `process(instance)` 处理单条样本
- 通过继承的 `run(dataset)` 处理数据集
- 通过 `operator_name` 与 `accelerator` 继承统一算子元信息
- 输出记录保留原字段，并追加标准化的采样结果字段

最小调用示例：

```python
from a3_sampling_operator import A3_Sampling_Operator

operator = A3_Sampling_Operator(
    sampling_mode="UNIFORM",
    num_frames=4,
)

sample = {
    "video_path": "demo.mp4",
    "video_id": "demo_001",
    "fps": 30,
    "duration_sec": 60.0,
    "total_frames": 1800,
    "segments": [
        {
            "segment_id": "seg_0001",
            "frame_range": {
                "start_frame": 0,
                "end_frame": 300
            }
        },
        {
            "segment_id": "seg_0002",
            "frame_range": {
                "start_frame": 300,
                "end_frame": 750
            }
        }
    ]
}

result = operator.process(sample)
samples = result["samples"]
```

## 12. 当前实现边界
这份实现已经能作为系统内 A3 的统一采样入口，但仍然是轻量参考实现。后续如果继续演进，优先补强的方向应当是：

- 为 `sample` 增加真实 `frame_uri` 或缓存句柄
- 引入图像质量过滤，避免模糊帧和纯黑帧
- 加入运动感知或变化感知采样
- 将采样模式与下游任务 profile 解耦
- 支持 A4 风格的自适应采样与重排
