# 上下文编排算子（A2）技术文档

## 1. 在系统中的定位
上下文编排算子位于 A1 划分之后、A3 采样之前。它不直接生成视觉证据，而是把离散片段整理成后续算子可稳定消费的上下文单元。

它解决的问题不是“这段视频值不值得看”，而是“这些片段应当如何组合、附着和裁剪，才能让后续采样与问答构建看到完整语义”。

## 2. 文献依赖等级

- 文献依赖等级：`P1 论文拆解`
- 最新支撑强度：`中`

### 2.1 文献锚点
- Panda-70M
- Shot2Story
- TimeChat-Captioner

### 2.2 从文献保留了什么
- 划分结果不应直接视为最终消费单元。
- 相邻片段常常需要 merge、attach 或 window 化处理。
- 后续采样与生成的质量，取决于上下文单元是否合理。

## 3. 子模式定义

需要强调的是，A2 的四个 mode 文献强度并不相同。它们更适合被理解为“同一类上下文编排需求下，不同强度的实现策略”。

### 3.1 强文献模式

#### MERGE_STITCH
把相邻短片段拼接成事件级单元。

文献依据：

- `Panda-70M` 的 split-and-stitch 思想最接近这一模式。
- 这类工作通常不是把第一次切分结果直接送入下游，而是允许后续对相邻片段做 event 级重组。

适用场景：

- A1 高召回切得较碎
- 事件往往跨越多个短片段
- 下游更关心完整事件，而不是极细粒度片段

### 3.2 弱文献模式

#### CONTEXT_ATTACH
为主片段附着前后文窗口，但不一定把它们真正 merge 成一个新事件。

文献依据：

- `Shot2Story` 这类多镜头叙事工作需要结合邻近镜头理解主镜头。
- `TimeChat-Captioner` 这类时间结构建模工作也隐含依赖前后文，而不是只看孤立片段。

它更像是对文献里“上下文一起看”的系统化抽象，而不是论文里单独命名的模块。

#### HIERARCHICAL_PACK
把粗粒度段落和细粒度子片段打包到同一上下文结构，让下游同时拿到“全局框架”和“局部锚点”。

文献依据：

- `TimeChat-Captioner` 这类方法往往同时保留场景级、事件级、时刻级信息。
- 一些长视频或叙事视频方法也会隐式使用 coarse-to-fine 的结构组织。

它的文献依据弱于 `MERGE_STITCH`，因为多数论文会直接把这种层次结构写进 prompt、caption schema 或中间脚本，而不会把它单独命名成一个算子 mode。

### 3.3 工程辅助模式

#### FILTER_HINT
对明显无效片段做软过滤标记，但不把 A2 定义成纯清洗器。

这个模式的定位主要是工程辅助：

- 它不是 A2 的学术核心卖点。
- 它的作用是保留旧“清洗”逻辑中合理但不应主导 A2 定义的部分。
- 更适合被看作 hint 或优先级标记，而不是硬性的删除机制。

## 4. HIERARCHICAL_PACK 的操作逻辑

`HIERARCHICAL_PACK` 的目标不是“合并片段”，而是“同时保留两个粒度”。

可以把它理解为以下三步：

### 4.1 先确定粗粒度单元
先从相邻片段中确定一个较大的语义范围，例如一个完整步骤、一个完整镜头段落、或一个完整事件区间。

例如：

- `seg_01`: 人打开冰箱
- `seg_02`: 人拿出牛奶
- `seg_03`: 人关上冰箱

这三个片段可以共同属于一个 coarse event：

- `pack_01`: 取牛奶

### 4.2 再保留细粒度子片段
不是把 `seg_01 ~ seg_03` 抹平成一个大段，而是保留它们各自的细粒度边界，让下游还能知道局部动作分别发生在哪里。

也就是说，pack 后的结构更像：

- 一个上层上下文单元
- 多个下层可引用子片段

### 4.3 下游按任务选择消费粒度
下游如果要做：

- 全局理解，就读 coarse pack
- 精细问答，就回到某个 child segment
- 多跳问题，就同时引用 coarse pack 和若干 child segment

所以它的本质是：

- `MERGE_STITCH` 更偏“把多个片段变成一个消费单元”
- `HIERARCHICAL_PACK` 更偏“把多个粒度一起打包保存”

### 4.4 它具体解决什么问题

它主要解决三类问题：

- 只看细粒度片段时，语义太碎，问题规划缺少全局框架
- 只看粗粒度片段时，答案锚点太粗，难以做 grounded QA
- 同一个视频同时需要 scene-level 和 moment-level 监督

### 4.5 一个简化示例

假设 A1 输出：

- `seg_10`: 教师打开 PPT
- `seg_11`: 展示标题页
- `seg_12`: 切到目录页
- `seg_13`: 开始讲第一节

`HIERARCHICAL_PACK` 可以组织成：

- `pack_03`:
  - coarse range: `seg_10 ~ seg_13`
  - summary: “课程导入与第一节开始”
  - children: `seg_10, seg_11, seg_12, seg_13`

这样：

- A7 如果要规划“这一段主要讲了什么”，可以直接读 `pack_03`
- A8 如果要生成“教师在哪一页开始讲第一节”的问题，可以落回 `seg_12` 或 `seg_13`

### 4.6 什么时候不该用

如果你的任务只需要：

- 固定多帧采样
- 简单 caption
- 不强调多粒度问答

那 `HIERARCHICAL_PACK` 往往没有必要，会增加结构复杂度。

## 5. 输入规范
消费 `VideoPacket.segments`，并可按需读取字幕、ASR 或已有元数据。

### 5.1 必需输入
- `segments[].segment_id`
- `segments[].timestamp_range`
- `segments[].frame_range`

### 5.2 可选输入
- `config.merge_gap_sec`
- `config.context_window_sec`
- `config.max_context_segments`
- `config.attach_text_context`

## 6. 输出规范
仍写回 `segments[]`，但会补充上下文字段：

- `context_group_id`
- `parent_segment_ids`
- `context_window`
- `hierarchy_level`
- `child_segment_ids`
- `orchestration_reason`
- `status`

## 7. 关键参数

| 参数 | 作用 |
|---|---|
| `merge_gap_sec` | 允许拼接的片段间隔 |
| `context_window_sec` | 为主片段附着的时间窗口 |
| `max_context_segments` | 单个上下文单元允许包含的片段数 |
| `pack_strategy` | 指定是否启用层次打包 |
| `attach_text_context` | 是否同步附着字幕/ASR 上下文 |

## 8. 与上下游的绑定关系

### 8.1 上游
- A1 划分算子

### 8.2 下游
- A3 采样算子优先消费已编排好的上下文单元
- A4、A6 可以直接复用 A2 输出的上下文边界

## 9. 典型失败模式
- 过度合并，导致上下文泄漏
- 上下文窗口过窄，关键因果关系被截断
- 层次打包过度，导致 coarse 和 fine 粒度同时失真
- 不同叙事层级混在一起，导致后续问答目标模糊

## 10. 推荐使用场景
- 长视频和多镜头叙事视频默认开启
- 需要做 grounded QA 或时间结构问答时强烈建议开启
- 只做最小基线时可退化为轻量 merge 模式
