# 上下文编排算子（A2）技术文档

## 1. 在系统中的定位
上下文编排算子位于 A1 划分之后、A3 采样之前。它不直接生成视觉证据，而是把离散片段整理成后续算子可稳定消费的上下文单元。

它解决的问题不是“这段视频值不值得看”，而是“这些片段应当如何组合、附着和裁剪，才能让后续采样与问答构建看到完整语义”。

## 2. 文献依赖等级

- 文献依赖等级：`P1 论文拆解`
- 最新支撑强度：`中`

### 2.1 文献锚点
- Panda-70M
- Shot2Story
- TimeChat-Captioner

### 2.2 从文献保留了什么
- 划分结果不应直接视为最终消费单元。
- 相邻片段常常需要 merge、attach 或 window 化处理。
- 后续采样与生成的质量，取决于上下文单元是否合理。

## 3. 子模式定义

### 3.1 MERGE_STITCH
把相邻短片段拼接成事件级单元。

### 3.2 CONTEXT_ATTACH
为主片段附着前后文窗口。

### 3.3 HIERARCHICAL_PACK
把粗粒度段落和细粒度子片段打包到同一上下文结构。

### 3.4 FILTER_HINT
对明显无效片段做软过滤标记，但不把 A2 定义成纯清洗器。

## 4. 输入规范
消费 `VideoPacket.segments`，并可按需读取字幕、ASR 或已有元数据。

### 4.1 必需输入
- `segments[].segment_id`
- `segments[].timestamp_range`
- `segments[].frame_range`

### 4.2 可选输入
- `config.merge_gap_sec`
- `config.context_window_sec`
- `config.max_context_segments`
- `config.attach_text_context`

## 5. 输出规范
仍写回 `segments[]`，但会补充上下文字段：

- `context_group_id`
- `parent_segment_ids`
- `context_window`
- `orchestration_reason`
- `status`

## 6. 关键参数

| 参数 | 作用 |
|---|---|
| `merge_gap_sec` | 允许拼接的片段间隔 |
| `context_window_sec` | 为主片段附着的时间窗口 |
| `max_context_segments` | 单个上下文单元允许包含的片段数 |
| `attach_text_context` | 是否同步附着字幕/ASR 上下文 |

## 7. 与上下游的绑定关系

### 7.1 上游
- A1 划分算子

### 7.2 下游
- A3 采样算子优先消费已编排好的上下文单元
- A4、A6 可以直接复用 A2 输出的上下文边界

## 8. 典型失败模式
- 过度合并，导致上下文泄漏
- 上下文窗口过窄，关键因果关系被截断
- 不同叙事层级混在一起，导致后续问答目标模糊

## 9. 推荐使用场景
- 长视频和多镜头叙事视频默认开启
- 需要做 grounded QA 或时间结构问答时强烈建议开启
- 只做最小基线时可退化为轻量 merge 模式
