# 上下文编排算子（A2）技术文档

## 1. 在系统中的定位
A2 位于 A1 划分之后、A3 采样之前，是默认主链路里的低成本前处理层。

它的职责被进一步收紧为三个功能：

- `MERGE_STITCH`
- `WINDOW_ATTACH`
- `FILTER_HINT`

因此，A2 只处理“局部、便宜、确定性强”的问题，例如：

- 相邻超短片段在短时间间隔内的轻量拼接
- 为主片段补固定前后文窗口
- 对过短、重复或空白片段做标记或过滤

A2 不负责高语义理解，也不负责层级骨架构建。它不回答“这些片段语义上是否属于同一事件”，也不回答“是否应该组织成 `event tree` / `action graph` / `query scaffold`”。这些工作应交给 A4、A5、A7。

## 2. 文献依赖等级

- 文献依赖等级：`P1 论文拆解`
- 最新支撑强度：`中`

### 2.1 文献锚点
- Panda-70M
- Shot2Story
- TimeChat-Captioner

### 2.2 从文献保留了什么
- 初次划分结果通常不是最终消费单元
- 下游常常需要局部 stitch 或固定窗口上下文，而不是只看孤立片段
- 这些步骤在文献中往往表现为预处理或数据脚本的一部分，而不是独立的高语义模块

## 3. 职能边界

### 3.1 A2 必做什么
- `MERGE_STITCH`：仅基于时间邻接、时长阈值和短间隔规则做局部 stitch
- `WINDOW_ATTACH`：按 `N` 个相邻片段或固定秒数补前后文
- `FILTER_HINT`：识别 `too_short`、`near_duplicate`、`blank_text` 等显性问题，并做软过滤标记

### 3.2 A2 明确不做什么
- 不做语义事件识别或主题归并
- 不做 `event tree`、`action graph`、`query scaffold` 这类高语义结构搭建
- 不做 caption、summary 或 evidence 生成
- 不根据模型语义相似度决定 pack、anchor 或层级
- 不处理视频边界正则化、顺序修正或额外边界清理逻辑

### 3.3 与 A4 / A5 / A7 的职责交接

| 算子 | 负责什么 | 为什么不放在 A2 |
|---|---|---|
| A4 多粒度描述 | 生成 frame / event / segment / context 描述 | 需要视觉或文本语义建模 |
| A5 时序结构抽取 | 识别事件链、阶段关系、时间窗 | 需要显式时序语义抽取 |
| A7 结构骨架构建 | 构造 narration chunk、event tree、action graph、query scaffold | 需要汇合多路语义构件 |

## 4. 子模式定义

需要强调的是，A2 的 mode 不是等强度的文献模块，而是围绕“轻量前处理”这个目标设计的不同实现策略。

### 4.1 强文献模式

#### MERGE_STITCH
把时间上紧邻、长度过短、明显被 A1 切碎的片段按硬规则重新拼接成更稳定的消费单元。

文献依据：

- `Panda-70M` 的 split-and-stitch 思想最接近这一模式
- 其重点是修复过碎切分，而不是重新做一轮完整事件理解

这里要特别区分 A1 和 A2：

- A1 负责给出初始切分边界或事件候选
- A2 的 `MERGE_STITCH` 只是对明显过碎的结果做局部修补

如果需要判断“这些片段是否真正构成同一事件”，那已经超出 A2，应交给 A5 或 A7。

适用场景：

- A1 高召回切分偏碎
- 相邻片段间隔很小，但单段长度太短，不利于采样和描述
- 你需要的是更稳定的消费单元，而不是新的语义层级

### 4.2 弱文献模式

#### WINDOW_ATTACH
为主片段附着固定前后文窗口，但不把这些片段声明为同一事件，也不构造上层层级单元。

文献依据：

- `Shot2Story` 这类多镜头叙事工作常常需要结合邻近镜头理解当前片段
- `TimeChat-Captioner` 这类链路也隐含依赖局部前后文，而不是只看孤立片段

更准确地说，这里保留的是“上下文共同消费”这个事实，而不是把“语义附着”本身当成 A2 的职责。

适用场景：

- 单片段语义不完整，前后镜头才构成可读上下文
- narration、教程、演示类视频存在明显的局部连续性
- 你只想补邻域信息，不想提前做高成本语义归并

### 4.3 工程辅助模式

#### FILTER_HINT
对明显异常片段做软过滤标记，避免它们继续占用预算。

它的定位是工程辅助：

- 保留合理的清洗逻辑
- 记录“为什么这个片段不值得继续消费”
- 但不把 A2 定义成纯删除器

## 5. 为什么不承担高语义骨架构建

`event tree`、`action graph`、`query scaffold` 已不再属于 A2。

原因很直接：

- 它们通常需要先判断哪些片段在语义上属于同一事件、同一动作链或同一问题证据域
- 它们还需要决定节点、关系和引用如何组织
- 这已经不再是“低成本前处理”，而是显式的语义结构搭建

继续把它留在 A2，会产生三个问题：

- 计算开销前置，入口层过重
- 与 A5 时序结构抽取的事件组织职责重叠
- 与 A7 结构骨架构建的高语义收敛职责重叠

因此，当前系统把这些高语义结构明确归到 A7，由 A4 / A5 / A6 提供语义构件后再构造。

## 6. 输入规范
消费 `VideoPacket.segments`，并可按需读取少量低成本元数据。

### 6.1 必需输入
- `segments[].segment_id`
- `segments[].timestamp_range`
- `segments[].frame_range`
- `segments[].duration_sec`

### 6.2 可选输入
- `config.merge_gap_sec`
- `config.max_merge_span_sec`
- `config.context_window_sec`
- `config.max_context_segments`
- `config.min_valid_duration_sec`
- `config.cleanup_rules`

## 7. 输出规范
仍写回 `segments[]`，但只补充轻量编排字段：

- `stitch_group_id`
- `context_window_refs`
- `cleanup_flags`
- `orchestration_reason`
- `status`

这些字段的含义应明确限制为：

- `stitch_group_id`：表示基于邻接规则得到的轻量拼接组，不等同于语义事件 id
- `context_window_refs`：表示固定前后文引用，不等同于高语义上下文单元
- `cleanup_flags`：记录片段被过滤或降优先级的原因

## 8. 关键参数

| 参数 | 作用 |
|---|---|
| `merge_gap_sec` | 允许拼接的最大时间间隔 |
| `max_merge_span_sec` | 单次 stitch 后允许的最大总时长 |
| `context_window_sec` | 为主片段补充的固定时间窗口 |
| `max_context_segments` | 单个窗口最多附着的相邻片段数 |
| `min_valid_duration_sec` | 判定过短或异常片段的下限 |
| `cleanup_rules` | 启用哪些 `FILTER_HINT` 规则 |

## 9. 与上下游的绑定关系

### 9.1 上游
- A1 划分算子

### 9.2 下游
- A3 采样算子优先消费经清洗和轻量编排后的片段
- A4 多粒度描述算子可复用 `stitch_group_id` 和 `context_window_refs`
- A5、A7 从这里接手更高语义的时序组织和骨架构建

## 10. 典型失败模式
- 只按时间间隔拼接，导致本应分开的事件被误合并
- 固定窗口设得过宽，导致采样预算被局部上下文吞噬
- 清洗规则过强，把稀有但有价值的片段提前过滤掉
- 试图用 A2 解决语义层级问题，导致入口层成本和职责同时失控

## 11. 推荐使用场景
- 长视频链路中的默认轻量前处理
- A1 切分结果较碎、较噪时
- 需要先把异常片段和局部窗口整理好，再进入 A3 / A4 / A5 / A7 时

如果任务明确需要：

- 事件级结构
- event tree / action graph
- 语义锚点

则不应继续扩张 A2，而应直接使用 A4、A5、A7。
