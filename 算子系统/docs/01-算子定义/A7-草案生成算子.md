# 草案生成算子（A7）技术文档

## 1. 在系统中的定位
草案生成算子位于当前默认主链路的前半段。它负责在 `A2 + A3` 给出的上下文单元和粗采样结果上，先生成一版**非最终态**的语义草案。

这里的“草案”不是最终 QA，而是供后续 grounding 回看使用的低成本候选，包括：

- 粗 caption
- question draft
- answer draft
- claim slots
- query seeds
- answer contract

A7 的核心作用是先把“这段视频大概值得问什么、答案大概会落在哪些 claim 上”结构化写出来，再交给 A8 把这些草案改写成可执行的 grounding query。

因此，A7 允许在 full grounding 之前生成文本，但它生成的是 `draft`，不是最终可接受样本。

## 2. 文献依赖等级

- 文献依赖等级：`P1 论文拆解`
- 最新支撑强度：`中强`

### 2.1 文献锚点
- Pinpointing Trigger Moment for Grounded Video QA
- LongViTU
- VDC-Agent
- TimeChat-Captioner
- Video-ChatGPT

### 2.2 从文献保留了什么
- 低成本全局理解往往先产出 caption、reasoning 或 QA draft，而不是一开始就做最重的 grounding。
- 生成式数据构建需要先给出“候选问题和候选答案”，再决定是否值得回看视频。
- 高质量链路需要把草案拆成可核验的 claim，而不是只保留自然语言整句。

## 3. 子模式定义

### 3.1 COARSE_CAPTION_DRAFT
先生成粗粒度 caption，再从 caption 中抽取候选 question/answer 线索。

适用场景：

- 教学视频
- 长视频概览
- 需要先快速覆盖样本池

### 3.2 QA_DRAFT
直接生成 question draft 和 answer draft，并同步补写基础 claim slots。

适用场景：

- 明确以 grounded QA 数据构建为目标
- 希望快速得到候选问答并进入后续回看

### 3.3 SLOT_ENRICHED_DRAFT
除 question/answer 草案外，还显式写出可供 A8 消费的 claim slots、query seeds 和预算提示。

这是当前系统推荐的默认模式。

### 3.4 LIGHT_BASELINE
只保留粗 caption 和最少量 question hints，用于极低成本链路或基础基线。

## 4. 输入规范
消费 `segments[]`、`samples[]`，可按需读取 A2 输出的上下文编排字段。

### 4.1 必需输入
- `segments[].segment_id`
- `samples[].sample_id`
- `samples[].timestamp_sec`
- `samples[].frame_index`

### 4.2 可选输入
- `segments[].context_group_id`
- `segments[].child_segment_ids`
- `config.draft_mode`
- `config.max_drafts_per_segment`
- `config.answer_type_set`
- `config.require_claim_slots`
- `config.allow_rejection_draft`

## 5. 输出规范
写入 `draft_units[]`。

每条 `draft_unit` 至少应包含：

- `draft_id`
- `source_segment_ids`
- `source_sample_ids`
- `coarse_caption`
- `question_draft`
- `answer_draft`
- `claim_slots`
- `answer_contract`
- `uncertainty_points`
- `budget_hint`
- `rejection_policy`
- `status`

其中 `claim_slots[]` 建议至少包含：

- `claim_id`
- `claim_text`
- `answer_role`
- `evidence_required`
- `query_seeds.event_hint`
- `query_seeds.entity_hint`
- `query_seeds.attribute_hint`
- `query_seeds.text_hint`

推荐的 `draft_unit` 结构如下：

## 6. 关键参数

| 参数 | 作用 |
|---|---|
| `draft_mode` | 选择草案生成模式 |
| `max_drafts_per_segment` | 每个片段最多生成多少草案 |
| `answer_type_set` | 控制输出答案类型 |
| `require_claim_slots` | 是否强制为每个草案补齐 claim slots |
| `allow_rejection_draft` | 是否允许直接写入“应拒答”的草案 |

## 7. 与上下游的绑定关系
- 上游：A2 提供稳定上下文，A3 提供粗采样结果
- 下游：A8 将草案改写成 query bundles
- 下游：A9 在 evidence revision 阶段可回读草案与 claim slots，检查哪些内容被保留、哪些内容被删除

## 8. 典型失败模式
- `question_draft` 很顺，但 `claim_slots` 没拆清楚，导致后续 query 不稳定
- `answer_draft` 写得过满，把还没证实的细节提前写死
- `coarse_caption` 和 `question_draft` 来自不同粒度，导致问题与答案锚点错位
- 草案没有显式写出 `rejection_policy`，导致后续系统更倾向强行修补而不是拒答

## 9. 推荐使用场景
- 默认的 grounded QA 数据构建主链路
- 希望先低成本覆盖大量视频样本，再选择性回看
- 需要把粗 caption、候选 QA 和 claim decomposition 串成同一中间态时
