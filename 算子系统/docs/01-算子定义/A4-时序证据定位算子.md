# 时序证据定位算子（A4）技术文档

## 1. 在系统中的定位
时序证据定位算子负责把 `A8` 产出的 query bundle 落到具体时间范围上，回答“到底应该回看哪一段、哪一个时刻、哪几个变化点”。

在当前落地版中，A4 的默认角色已经不是“先生成一个通用时序证据池”，而是：

- 路线一中的 `post-hoc temporal grounding` 回看节点
- 路线二中的 `question-conditioned temporal localization` 入口节点

因此，A4 的默认工作方式是**被 query 驱动**，而不是无条件先行。

## 2. 文献依赖等级

- 文献依赖等级：`P2 论文原生`
- 最新支撑强度：`很强`

### 2.1 文献锚点
- Video-in-the-Loop
- Can I Trust Your Answer? Visually Grounded Video Question Answering
- LeAdQA
- ED-VTG
- Think with Grounding
- TimeChat-Captioner

### 2.2 从文献保留了什么
- 高质量 grounded QA 需要显式定位关键 span，而不是只保留全局视频印象。
- 长视频处理常常先做 temporal localization，再决定是否继续触发更贵的空间或文本分支。
- 时序证据不仅可以支持答案，也可以反驳草案中的错误 claim。

## 3. 子模式定义

### 3.1 DRAFT_CONDITIONED_SPAN
消费 A7 草案和 A8 query bundle，优先收缩与 `must_support_claims` 对应的时间范围。

这是路线一中的默认模式。

### 3.2 EXTERNAL_QUESTION_SPAN
直接消费外部问题，经 A8 改写后执行时序定位。

这是路线二中的标准模式。

### 3.3 CHANGE_TRACE
对变化点、转折点、开始/结束时刻进行更细粒度追踪。

### 3.4 MULTI_EVIDENCE_PRELOCATE
先给 A5/A6 缩小时序工作区间，作为多证据分支的前置约束。

## 4. 运行前提与工作层次

### 4.1 默认前提
A4 至少需要以下条件：

- A2 已经给出相对稳定的上下文边界
- A3 已经给出可消费的采样结果
- A8 已经给出可执行的 query bundle，或系统已接收到外部问题并完成规范化

也就是说，当前系统默认不再把“无 query 的通用时序池”作为 A4 的首要形态。

### 4.2 三种工作层次

#### 第一层：草案驱动的 span 缩小
系统先有 A7 的草案，再由 A8 规范化，A4 负责找到最可能支撑或推翻 claim 的关键 span。

#### 第二层：外部问题驱动的时序定位
当系统已有 question 时，A4 直接围绕该 question 的 query bundle 做 temporal grounding。

#### 第三层：弱先验辅助定位
在没有明确 question 的研究型设置中，A4 仍可用任务先验或事件先验做弱定位，但这不再是当前网站默认主链路。

## 5. 输入规范
消费 `segments[]`、`samples[]`、`query_bundles[]`，必要时可读取 A2 的上下文边界和 A7 的 claim slots。

### 5.1 必需输入
- `samples[].sample_id`
- `samples[].timestamp_sec`
- `segments[].segment_id`
- `query_bundles[].bundle_id`
- `query_bundles[].temporal_query`

### 5.2 可选输入
- `draft_units[].claim_slots`
- `config.max_events_per_bundle`
- `config.localization_granularity`
- `config.include_change_summary`
- `text_signals[]`
- `regions[]`

## 6. 输出规范
写入 `temporal_events[]`，推荐字段：

- `event_id`
- `segment_id`
- `query_bundle_id`
- `start_sec`
- `end_sec`
- `event_type`
- `summary`
- `supporting_claim_ids`
- `event_role`
- `trigger_sample_ids`
- `span_score`

其中建议显式区分以下语义：

- `event_role=support`：该 span 支撑某个 claim
- `event_role=refute`：该 span 与某个 claim 冲突
- `event_role=context`：该 span 提供上下文但不单独决定答案

## 7. 关键参数

| 参数 | 作用 |
|---|---|
| `max_events_per_bundle` | 每个 query bundle 最多保留多少关键 span |
| `localization_granularity` | 定位粒度，如 coarse 或 fine |
| `include_change_summary` | 是否额外输出变化摘要 |
| `temporal_first_policy` | 是否强制先缩 span 再触发 A5/A6 |

## 8. 与上下游的绑定关系
- 上游：A3 提供采样结果，A8 提供 query bundles
- 上游补充：A7 的 claim slots 可作为定位目标
- 下游：A5 和 A6 通常在 A4 缩小后的 span 内继续取证
- 下游：A9 直接消费 A4 输出的时序证据，做答案回修和严格审查

## 9. 典型失败模式
- 定位范围过宽，导致 A5/A6 的工作成本失控
- 只找支持证据，不保留反证或冲突证据
- 复合问题没有先拆 claim，导致 A4 输出的 span 过于模糊
- 事件拆分过细，导致后续答案回修没有稳定时序锚点

## 10. 推荐使用场景
- 当前默认 grounded QA 数据构建主链路
- 已有 question 的 benchmark 推理链路
- 强时间顺序依赖和关键时刻判定任务
